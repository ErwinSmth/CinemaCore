# Movie Service (Gestión de Cartelera)

## 1. Responsabilidad
Gestionar el catálogo de películas del cine, permitiendo al administrador buscar títulos globales usando la API de TMDB, curar la información a nivel local y publicarla para los clientes. Provee alta velocidad de lectura para el Front-end sin sobrecargar la base de datos o el ancho de banda del backend.

## 2. Flujo de Gestión de Cartelera (Integración TMDB)

### Fase 1: La Búsqueda Externa (El Administrador explora)
*   **Intención:** El Administrador del cine ingresa a su panel privado y navega a "Agregar Nueva Película".
*   **Acción:** Ingresa el nombre de la película (ej. "Obsesión").
*   **Por debajo:** El Frontend envía esta palabra al Backend. El Backend actúa como puente y pregunta al catálogo mundial (TMDB). TMDB le devuelve una lista general de coincidencias que se reenvía al Frontend.
*   **Resultado Visual:** Una cuadrícula con varias opciones de películas que coinciden con el nombre, mostrando año y portada básica.

### Fase 2: La Selección y Curación Local (El Administrador personaliza)
*   **Acción:** El Administrador identifica la película y hace clic en "Importar al Cine".
*   **Por debajo:** El Backend va a TMDB y descarga toda la información específica (elenco, portadas, trailers).
*   **Resultado Visual (Formulario de Curación):** El Administrador ve un formulario pre-llenado.
    *   Título y sinopsis escritos automáticamente (editables).
    *   Selecciona Póster Oficial y Trailers.
    *   **Aporte Local (Crucial):** Elige Restricción de Edad (APT, +14, +18) y define Estado ("PRE-ESTRENO" o "CARTELERA").

### Fase 3: La Persistencia (Guardado en `db_movies`)
*   **Acción:** Clic en "Guardar en Base de Datos".
*   **Por debajo:** El Frontend empaqueta las decisiones finales y las envía al Backend.
*   **Impacto en BD:** Los datos de búsqueda rápida (título, duración, estado, edad) van a columnas relacionales tradicionales. La información variable y multimedia (urls de imagen, trailers, actores) se comprime y **guarda en una columna JSONB**. A partir de aquí, el cine es dueño de su información.

### Fase 4: La Visualización del Cliente (Consumo Optimizado)
*   **Intención:** Un Cliente final ingresa para ver la cartelera.
*   **Por debajo:** El Frontend pide al Backend únicamente las películas con estado "CARTELERA".
*   **Respuesta Rápida:** El Backend hace una consulta rápida, toma el JSONB y lo envía al cliente. *No envía imágenes pesadas*, solo texto y enlaces.
*   **Resultado Visual:** El dispositivo del cliente descarga las imágenes directamente desde los servidores de TMDB usando la conexión del usuario, manteniendo el Backend relajado.

## 3. Integración API de TMDB (Documento Técnico)

### Fase 1: Petición de Búsqueda (Search)
*   **Endpoint:** `GET https://api.themoviedb.org/3/search/movie`
*   **Query Params:** `api_key={TU_CLAVE}&query=obsesion&language=es-MX`
*   **Dato Necesario:** Solo se necesita capturar el `id` de TMDB del arreglo `results` para pasar a la siguiente fase.

### Fase 2: Petición de Detalle Completo (Details + Append)
*   **Endpoint:** `GET https://api.themoviedb.org/3/movie/{id}`
*   **Query Params:** `api_key={TU_CLAVE}&language=es-MX&append_to_response=credits,videos,images`
*   **Mapeo Exacto de Datos a Extraer:**
    *   `id` (Número): Identificador único de TMDB.
    *   `title` (String): Título oficial.
    *   `overview` (String): Sinopsis.
    *   `runtime` (Número): Duración en minutos (vital para Showtime Service).
    *   `release_date` (String): Fecha de estreno (YYYY-MM-DD).
    *   `credits.cast[].name` (String): Nombres de actores principales (tomar los primeros 3 a 5).
    *   `credits.crew[].name` (String): Director (Filtrar objeto donde `job` == "Director").

### Fase 3: Reconstrucción de Recursos Multimedia
El Backend debe armar las URLs de los medios antes de entregarlas (o guardarlas):

**A. URL de Portadas (Imágenes)**
*   **Fuente:** `images` -> `posters` -> extraer `file_path`.
*   **Fórmula:** `[BASE_URL] + [TAMAÑO] + [FILE_PATH]`
*   **Ejemplo:** `https://image.tmdb.org/t/p/` + `w500` + `/rmCkNtzYR2xTOO3ZXmIqB5zgYdE.jpg`

**B. URL de Trailers (Videos)**
*   **Fuente:** `videos` -> `results`.
*   **Filtro Obligatorio:** Iterar y mantener objetos donde `site` == "YouTube" y `type` == "Trailer".
*   **Fórmula:** `[URL_YOUTUBE] + [KEY]`
*   **Ejemplo:** `https://www.youtube.com/watch?v=` + `1mTjfMFyPi8`

## 4. Data Model Specification (`db_movies`)

El diseño de la base de datos utiliza un enfoque híbrido que mezcla el modelo relacional tradicional (`peliculas` y `restriccion_edad`) con el almacenamiento de documentos estructurados (`metadata JSONB`).

## 5. API Contract (Para API Gateway y Frontend)

### `GET /api/v1/movies/search?query={titulo}`
*   **Actor:** Administrador
*   **Propósito:** Busca en TMDB y devuelve resultados resumidos.
*   **Respuesta Exitosa (200 OK):** Lista de películas (id, título, año).

### `GET /api/v1/movies/tmdb/{tmdbId}`
*   **Actor:** Administrador
*   **Propósito:** Obtiene el detalle completo de TMDB (Fase 2) para el formulario de curación.
*   **Respuesta Exitosa (200 OK):** Objeto detallado (título, sinopsis, cast, urls).

### `POST /api/v1/movies`
*   **Actor:** Administrador
*   **Propósito:** Guarda la película curada en la base de datos local.
*   **Payload Esperado:**
    ```json
    {
      "tmdbId": 12345,
      "title": "Obsesión",
      "runtime": 120,
      "ageRating": "+14",
      "status": "CARTELERA",
      "details": {
         "overview": "...",
         "poster_url": "..."
      }
    }
    ```
*   **Respuesta Exitosa (201 Created):** Objeto de la película guardada con su UUID interno.

### `GET /api/v1/movies?status=CARTELERA`
*   **Actor:** Cliente Final
*   **Propósito:** Obtener la cartelera para mostrar al cliente.
*   **Respuesta Exitosa (200 OK):** Lista de películas activas. Consulta ultra rápida a BD usando filtro por `status`.

## 6. Acceptance Criteria (TDD Scenarios)

*   **Scenario 1: Búsqueda exitosa en TMDB**
    *   **Given** que el administrador provee un término de búsqueda válido
    *   **When** se invoca `/api/v1/movies/search`
    *   **Then** debe retornar una lista de resultados mapeados correctamente y HTTP 200.
*   **Scenario 2: Guardado de película nueva**
    *   **Given** un payload válido con un `tmdbId` que no existe en BD
    *   **When** se hace POST a `/api/v1/movies`
    *   **Then** debe persistirse en PostgreSQL, incluyendo el campo JSONB, y retornar HTTP 201.
*   **Scenario 3: Intento de guardar película duplicada**
    *   **Given** una película con un `tmdbId` que ya existe en la base de datos
    *   **When** se hace POST a `/api/v1/movies`
    *   **Then** debe lanzar una excepción de restricción de unicidad y retornar HTTP 409 Conflict.
