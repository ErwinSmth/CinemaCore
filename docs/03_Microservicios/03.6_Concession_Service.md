# Concession Service (Gestión de Dulcería) - DRAFT (Fase 2)

## 1. Responsabilidad
Gestionar el inventario, catálogo y carritos de compra virtuales correspondientes a las comidas y bebidas (Dulcería) del cine. Este microservicio actúa como un módulo de e-commerce separado de la venta de butacas, permitiendo que la compra de alimentos siga reglas de negocio independientes a las de los tickets.

## 2. Decisiones Arquitectónicas Clave

### Separación de Responsabilidades (Loose Coupling)
**NO** se debe mezclar la lógica de la comida con el `Seat Service` ni con el `Movie Service`. El `Seat Service` enfrenta un alto nivel de concurrencia y carrera de condiciones (*Race Conditions*) debido al Optimistic Locking de las butacas. La comida es un e-commerce tradicional. Mantenerlos separados permite escalar el sistema de dulcería sin afectar la venta crítica de tickets.

### Estrategia de Inventario (El Patrón de Stock Híbrido)
Para reflejar la realidad operativa de un cine, el servicio manejará dos tipos de comportamiento de stock mediante una bandera en la tabla de productos (`maneja_stock`: BOOLEAN):

1.  **Stock Infinito (Preparados Continuos):**
    *   **Aplicación:** Popcorn, Gaseosas de máquina, Nachos.
    *   **Comportamiento:** `maneja_stock = false`. El sistema asume que la producción es continua y no descuenta inventario por cada venta. La única restricción es un "Kill-Switch" manual del administrador que cambia el estado a `AGOTADO` en caso de avería en maquinaria.
2.  **Stock Estricto (Empaquetados y Coleccionables):**
    *   **Aplicación:** Vasos temáticos de películas (ej. "Vaso Deadpool"), Merchandising limitado, Chocolates importados.
    *   **Comportamiento:** `maneja_stock = true`. El sistema exige una validación dura (`stock_actual > 0`). Si el usuario lo añade al carrito, se ejecuta un bloqueo temporal (reserva) similar a una butaca. Si el usuario no concreta el pago en 10 minutos, un proceso asíncrono devuelve la unidad al stock disponible.

## 3. Integración en el Flujo de Compra (Checkout)

La integración final se dará mediante el Patrón SAGA (comunicación asíncrona) que rige las reglas globales del monorepo:

1.  El `Seat Service` asegura las butacas.
2.  El `Concession Service` provee los items del carrito.
3.  Un futuro *Order/Payment Service* (o Gateway) sumará el costo del Ticket + Dulcería y enviará un único cobro a la pasarela de pagos (Culqi).
4.  Si el pago falla, la SAGA emitirá un evento compensatorio que liberará la butaca y devolverá el stock de cualquier merchandising bloqueado.

## 4. Endpoints Proyectados

### Administrador (Gestión de Inventario)
- `POST /concessions/products`: Crear nuevo ítem (Sube foto, nombre, precio y define si maneja stock duro).
- `POST /concessions/combos`: Agrupar IDs de productos existentes bajo un nuevo precio promocional.
- `PUT /concessions/products/{id}/status`: El "Kill-Switch" para agotar productos rápidamente.

### Cliente (Consumo)
- `GET /concessions`: Lista los productos y combos disponibles para ser mostrados en la UI del Frontend.
