# Showtime Service (Programación de Funciones)

## 1. Responsabilidad
Administrar la programación física del cine: horarios, asignación de salas, formatos de proyección (2D, 3D) y precios base por función. Actúa como el nexo logístico entre el catálogo global (`Movie Service`) y el inventario físico (`Seat Service`).

## 2. Dominio y Reglas de Negocio

### 2.1 Patrón Orquestador (Orchestrator Pattern) en Creación de Funciones
*   **Regla Crítica:** Cuando un Administrador programa una nueva Función (Showtime), el `Showtime Service` actúa como el **Orquestador de la transacción**. Guarda el registro en su base de datos y **obligatoriamente** debe dar una orden síncrona al `Seat Service` para que pre-genere los tickets físicos de la sala.
*   **Calculo de Horarios:** El sistema debe usar la `duracion_min` de la película (obtenida del `Movie Service` o enviada desde el Frontend) para calcular automáticamente la `fecha_fin` de la función, asegurándose de evitar solapamientos (overlaps) en la misma sala.

### 2.2 Desacoplamiento de Entidades
*   El `Showtime Service` **no** guarda el nombre de la película ni sus pósters. Guarda únicamente el `movie_id` como referencia. El diseño y composición de la cartelera final recaerá en el Frontend o en el API Gateway.

## 4. API Contract (Endpoints Internos)

### `GET /api/v1/showtimes?fecha={yyyy-MM-dd}`
*   **Actor:** Cliente Final
*   **Propósito:** Listar las funciones disponibles para un día específico.
*   **Respuesta Exitosa (200 OK):** Lista de funciones (id, sala, hora de inicio, tipo de proyección, precio).

### `POST /api/v1/showtimes`
*   **Actor:** Administrador (Requiere `ROLE_ADMINISTRADOR`)
*   **Propósito:** Programar una nueva función en una sala.
*   **Payload Esperado:** `movie_id`, `sala_id`, `proyeccion_id`, `fecha_inicio`, `precio_ticket`.
*   **Respuesta Exitosa (201 Created):** Objeto de la función. El endpoint debe validar internamente conflictos de horario y mandar a pre-generar asientos antes de devolver 201.

## 5. Acceptance Criteria (TDD Scenarios)

*   **Scenario 1: Programación exitosa y Pre-generación**
    *   **Given** un horario libre en la `Sala 1`
    *   **When** el Admin hace POST a `/api/v1/showtimes`
    *   **Then** se crea la función en la DB, se realiza la petición HTTP exitosa al Seat Service para pre-generar butacas, y se retorna HTTP 201.
*   **Scenario 2: Solapamiento de horarios (Overlap)**
    *   **Given** una función ya existente en la `Sala 1` de 14:00 a 16:30
    *   **When** el Admin intenta programar otra función en la misma sala a las 15:00
    *   **Then** el sistema rechaza la operación por conflicto de horario y retorna HTTP 409 Conflict.
