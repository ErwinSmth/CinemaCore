# Seat Service (Inventario y Reservas)

## 1. Responsabilidad
Administrar la distribución física y el estado de cada asiento en tiempo real, garantizando la consistencia y previniendo colisiones durante el proceso de compra de boletos.

## 2. Estrategia de Concurrencia y Reserva de Asientos

### El Ciclo de Vida del Ticket (Pre-generación)
*   **Regla de Arquitectura:** Los registros en la tabla `tickets` NO se crean en el momento en que el usuario hace clic en un asiento.
*   **¿Cuándo se crean?:** Se pre-generan en bloque en el momento en que el Administrador crea la función (Showtime). Si la sala tiene 100 butacas, el sistema hace un `INSERT` de 100 filas en la tabla `tickets` automáticamente.
*   **Estado Inicial:** Todas las filas nacen con `user_id = null`, `status = 'AVAILABLE'` y `version = 0`.
*   **Beneficio:** Al pre-existir la fila, cuando múltiples usuarios intentan comprar al mismo tiempo, el sistema solo compite por hacer un `UPDATE`, permitiendo que el Optimistic Locking de JPA funcione a la perfección.

## 3. Anatomía de las Columnas Especiales (`tickets`)
La tabla de tickets posee 3 columnas encargadas de orquestar transacciones concurrentes:

### A. `status` (El Semáforo del Negocio)
*   **Definición:** Etiqueta de texto (`AVAILABLE`, `LOCKED`, `SOLD`).
*   **Uso:** Le indica al Frontend de qué color pintar la butaca (Verde, Amarillo, Gris). A la base de datos no le afecta directamente la concurrencia a través de este campo textual.

### B. `version` (El Escudo de BD - Optimistic Locking)
*   **Definición:** Contador numérico interno controlado nativamente por JPA/Hibernate (`@Version`).
*   **Uso:** Impide matemáticamente que dos transacciones modifiquen la misma fila al mismo tiempo (Lanza `OptimisticLockException` si las versiones no cuadran). Es invisible para el Frontend.

### C. `locked_until` (El Temporizador de Abandono)
*   **Definición:** Un campo de fecha y hora (`TIMESTAMP`).
*   **Uso:** Define el tiempo límite que un asiento puede permanecer en estado `LOCKED`. Si pasa de este tiempo (ej. pasaron los 5 minutos de gracia), el Backend puede devolver la butaca a estado `AVAILABLE`.

## 4. Caso Práctico de Colisión: Escenario de "Race Condition"

¿Qué sucede cuando dos personas intentan comprar la misma butaca exactamente al mismo tiempo?

**Minuto 0: El Escenario**
*   Usuario 1 y Usuario 2 ven el asiento "G-14" en color verde.
*   Frontend tiene en memoria: `Asiento G-14 | status: AVAILABLE | version: 1`.

**Minuto 1: El Choque**
*   Ambos hacen clic en "Comprar G-14" casi al unísono.

**Resolución en PostgreSQL + Spring Boot:**
1.  **Petición Usuario 1 (Llega primero):**
    *   Backend envía a BD: *"Actualiza G-14 a LOCKED, suma 5 min a locked_until, cambia versión a 2, SOLO SI la versión sigue siendo 1"*.
    *   BD responde: *1 Fila Actualizada*.
    *   Usuario 1 avanza a la pasarela de pagos.
2.  **Petición Usuario 2 (Llega milisegundos tarde):**
    *   Backend envía la misma orden: *"Actualiza G-14 a LOCKED, cambia versión a 2, SOLO SI la versión sigue siendo 1"*.
    *   BD busca versión 1, pero ya no la encuentra (Usuario 1 la cambió a 2).
    *   BD responde: *0 Filas Actualizadas*.
3.  **La Excepción:**
    *   Spring Boot detecta que se afectaron 0 filas y lanza `OptimisticLockException`.
    *   El Backend atrapa el error y le devuelve un mensaje amigable al Usuario 2: *"Lo sentimos, este asiento acaba de ser tomado por otro usuario"*.
