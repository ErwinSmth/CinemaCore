# Auth Service (Identidad y Seguridad)

## 1. Responsabilidad
El Auth Service es el guardián del sistema. Su única responsabilidad es gestionar el registro de usuarios, la validación de credenciales (login) y la emisión de tokens JWT. No debe contener lógica de negocio sobre cines, películas o asientos.

## 2. Stack Tecnológico
*   **Lenguaje:** Java 17
*   **Framework:** Spring Boot 3+
*   **Base de Datos:** PostgreSQL 15 (vía Spring Data JPA)
*   **Migraciones:** Flyway
*   **Seguridad:** Spring Security + JJWT (JSON Web Token)

## 3. Dependencias Inter-servicio
*   **Llamadas Salientes (Outbound):** Ninguna. Este servicio es completamente independiente y no consume APIs de otros microservicios.
*   **Llamadas Entrantes (Inbound):** Recibe peticiones exclusivamente desde el **API Gateway**.

## 4. Configuración y Variables de Entorno
> **Nota de Seguridad:** Nunca se deben colocar contraseñas o secretos reales en la documentación o repositorios. Solo se listan las claves requeridas para levantar el entorno.

| Variable de Entorno | Descripción | Valor por Defecto / Ejemplo |
| :--- | :--- | :--- |
| `SPRING_DATASOURCE_URL` | Cadena de conexión a la BD | `jdbc:postgresql://localhost:5433/db_auth` |
| `SPRING_DATASOURCE_USERNAME` | Usuario de la BD | `postgres` |
| `SPRING_DATASOURCE_PASSWORD` | Password de la BD | *(Secreto)* |
| `JWT_SECRET` | Llave secreta para firmar tokens | `una_clave_muy_larga_y_secreta_aqui` |
| `JWT_EXPIRATION` | Tiempo de vida del JWT en ms | `3600000` (1 hora) |

## 5. Estrategia de Seguridad (JWT y BCrypt)
*   **Contraseñas:** Jamás se guardan en texto plano. Se utiliza `BCryptPasswordEncoder` para hashear las contraseñas antes de guardarlas en la base de datos.
*   **Tokens JWT (Stateless):** Cuando el login es exitoso, se emite un JWT firmado.
*   **Secreto Compartido:** El secreto (clave de firma) del JWT debe ser el mismo que conoce el **API Gateway**. De esta manera, el API Gateway puede verificar la validez del token localmente sin hacer peticiones de red al Auth Service por cada request.
*   **Roles (RBAC):** El payload del JWT incluye el ID del usuario y su Rol (`ROLE_CLIENTE`, `ROLE_ADMINISTRADOR`, `ROLE_TAQUILLERO`), para que el API Gateway o los microservicios sepan qué permisos tiene.

## 6. Manejo de Errores y Excepciones (GlobalExceptionHandler)

El servicio implementa un manejador global de excepciones (`@RestControllerAdvice`) para asegurar que todos los errores se devuelvan al Frontend con un formato JSON estándar y predecible, evitando exponer trazas de error de Java (`stacktraces`).

### 6.1 Estructura del ErrorResponse (DTO)
Cada vez que ocurre una excepción, el sistema responde con este formato estandarizado:
```json
{
  "timestamp": "2026-06-24T10:15:30",
  "status": 400,
  "error": "Bad Request",
  "message": "Mensaje descriptivo del error",
  "path": "/auth/register"
}
```

### 6.2 Excepciones Capturadas
*   `MethodArgumentNotValidException`: Captura errores de validación de los DTOs (ej. campos vacíos). Retorna el mensaje de validación fallido con código `400 Bad Request`.
*   `BadRequestException`: Excepción personalizada para lógica de negocio (ej. "El correo electrónico ya está registrado"). Retorna `400 Bad Request`.
*   `BadCredentialsException`: Lanzada por Spring Security cuando la contraseña no coincide. Retorna "Credenciales inválidas" con `401 Unauthorized`.
*   `UsernameNotFoundException`: Cuando el email no existe en la base de datos. Retorna "Usuario no encontrado" con `404 Not Found`.
*   `Exception` (Genérica): Captura cualquier error no controlado. Retorna "Error interno del servidor" con `500 Internal Server Error`.

## 8. API Contract (Endpoints Internos)

> **Nota:** Estos endpoints son llamados por el API Gateway, quien expone la ruta `/api/v1/auth/` al Frontend.

### `POST /api/auth/register`
*   **Propósito:** Crear una nueva cuenta de cliente.
*   **Payload Esperado:**
    ```json
    {
      "email": "juan@example.com",
      "contrasena": "SecretPassword123",
      "nombres": "Juan",
      "apellidos": "Pérez"
    }
    ```
*   **Respuesta Exitosa (201 Created):** Retorna inmediatamente el token JWT y los datos del usuario para auto-loguearlo.
    ```json
    {
      "token": "eyJhbGciOiJIUzI1NiIsIn...",
      "email": "juan@example.com",
      "roles": ["ROLE_CLIENTE"]
    }
    ```
*   **Restricción:** Solo crea usuarios con rol `ROLE_CLIENTE`. Los administradores se deben crear directamente en base de datos o por otro medio interno.

### `POST /api/auth/login`
*   **Propósito:** Validar credenciales y devolver token.
*   **Payload Esperado:**
    ```json
    {
      "email": "juan@example.com",
      "contrasena": "SecretPassword123"
    }
    ```
*   **Respuesta Exitosa (200 OK):**
    ```json
    {
      "token": "eyJhbGciOiJIUzI1NiIsIn...",
      "email": "juan@example.com",
      "roles": ["ROLE_CLIENTE"]
    }
    ```
*   **Respuesta Error (401 Unauthorized):** Si la contraseña o email son incorrectos.

## 9. Acceptance Criteria (TDD Scenarios)

*   **Scenario 1: Login exitoso**
    *   **Given** un usuario registrado en la base de datos
    *   **When** se hace POST a `/auth/login` con credenciales correctas
    *   **Then** se genera un JWT válido firmado con el secreto, y retorna HTTP 200.
*   **Scenario 2: Contraseña incorrecta**
    *   **Given** un usuario registrado
    *   **When** se hace POST a `/auth/login` con contraseña incorrecta
    *   **Then** NO se revela si el usuario existe o no, se retorna "Credenciales inválidas" y HTTP 401.
*   **Scenario 3: Correo duplicado al registrar**
    *   **Given** un correo que ya existe en la tabla `users`
    *   **When** se hace POST a `/auth/register` con ese correo
    *   **Then** lanza `BadRequestException` y retorna HTTP 400 Bad Request con el mensaje "El correo electrónico ya está registrado".
